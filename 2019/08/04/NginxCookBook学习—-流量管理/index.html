NginxCookBook学习—-流量管理

发表于 2019-08-04 | 分类于 Nginx |
字数统计: 682 | 阅读时长 ≈ 2

介绍

NGINX和NGINX Plus也被归类为web流量控制器。您可以使用NGINX根据许多属性智能地路由流量和控制流量。本章将介绍NGINX根据百分比、利用客户端的地理位置拆分客户端请求以及以速率、连接和带宽限制的形式控制流量流。当您阅读本章时，你可以将这些特性混合并匹配起来使用以创造无数的可能性。

A/B Testing(分离测试)

问题:在一个文件或应用程序的两个或多个版本之间分割客户机，以测试验收

解决:使用split_clients模块将一部分客户端指向不同的上游池:

split_clients指令将您提供的字符串散列为第一个参数，and divides that hash by the percentages provided to map the value of a variable provided as the second parameter(该句 不是很懂，不知道是什么除以什么，有知道的大佬交流下).第三个参数是一个包含键-值对的对象，其中键是百分比权重，值是自己分配的值。键可以是百分比，也可以是星号。星号表示所有百分比计算完毕后的其余部分。$ variable变量的值将是backendv2(20%的客户端IP地址)和backendv1(剩下的80%)。

(官网解析示例：原始字符串(即 remote_addr)的值经过MurmurHash2算法进行了哈希。 示例中，哈希值从0到21474835（0.5%—对应上述百分比值）对应于变量$variant的".one"值， 哈希值从21474836到107374180（2%）对应于值".two"， 哈希值从107374181到4294967295对应于值""（一个空字符串）。)

在本例中，backendv1 和 backendv2 表示不同的上游服务器池。其可以集合 proxy_pass指令使用:

使用变量$variant，流量将在两个不同的应用服务器池之间分割。

这种类型的A/B测试是有用的，例如当在电子商务网站上测试不同类型的营销和前端功能的转化率时。应用程序通常使用一种名为canary release的部署类型。在这种类型的部署中，流量会缓慢地切换到新版本。在推出新版本的代码时，在应用程序的不同版本之间分割客户机非常有用，以便在出现错误时限制错误的影响范围。无论是出于什么原因要分割用户到两种不同类型的应用集合上，MGINXdoe可以通过split_clients模块使其变得更容易。
