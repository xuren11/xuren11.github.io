个人观点:万丈高楼平地起。任何技术的发展都是从 最基础最简单也是最繁琐的 方式做起。了解最基础的技术，也就能了解到 该基础技术的优缺点，同时也就能知道为什么会出现新技术，新技术的出现有解决了基础技术的哪些痛点，相信这样我们才能更好的理解新技术学习方技术。

传统–JDBC

步骤(概念)：DriverManager –> Connection –> Statement –> ResultSet

依赖: mysql-connector-java

一般将前两步(DriverManager[驱动类加载] Connection[连接获取/释放] 封装为一个 JDBCUtils类，使用配置文件的形式配置 数据库地址/用户/密码等信息，这样更利于修改维护)

开发

形同Hibernate。自己的Dao接口 继承 Repository<T,D>接口即可！

Repository 详解

Repository Hierarchy:

1.CrudRepository (内置 crud操作)
2.PagingAndSortingRepository (分页与排序)
3.JpaRepository

其他类：JpaSpecifictonExcutor

Repository接口 是 Spring Data的核心接口，不提供任何方法

Repository的子接口:

CrudRepository ：继承Repository，实现CRUD相关的方法

PagingAndSortingRepository： 继承C入度Repository，实现了分页排序 相关的方法

JpaRepository：继承PagingAndSortingRepository，实现JPA规范相关的方法

弊端： 方法名长， 复杂查询难

@Query注解使用

在Repository方法中使用，不需要遵循查询方法命名规则

只需要将@Query定义在Repository中之上的方法即可

支持 命名参数(@Param指定参数的名称) 及 索引参数(从前往后，参数顺序为 1 2 3 …….)的

支持本地查询

使用事务

@Modifying注解 结合 @Query注解执行 更新操作

@Transactional

在SpringData JPA中，对于 update操作，是必须要事务的！(读的时候，不需要事务)

事务在 SpringData中的使用：

1.事务一般在Service层

2.@Query @Modifying(两者共同使用在 dao层的方法上) @Transactional(使用在Service层的方法上) 要综合使用。

CrudRepository接口的使用

使用与Repository接口的使用一致！自己 实现类继承它即可！

PagingAndSortingRepository接口

该接口 包含排序和分页功能

带排序的查询：findAll(Sort sort)

带排序的分页查询：findAll(Pageable pageable)

常用Pageable实现类：*PageRequest *(就相当于 分页插件)【其中具体内容查看API即可】

JpaRepository接口

findAll() findAll(Sort sort) save(entities) flush deleteInbatch(entities)

JpaSpecificationExcutor接口

Specification封装了JPA Criteria查询条件，可以进行更复杂的查询

(可灵活构建条件，使用灵活)
